Relational Database (DB)

A relational database is a digital representation of a relational model of data (how information is linked together).
A typical DB stores the organized data into tables (or relations).
In a table, the different columns correspond to the fields (or attribute) of our data and the rows correspond to the different records (or tuples).

A practical example

Say we want to create a database in order to classify galaxies by morphology like in Figure 2.

A galaxy is a system comprising stars (like our Sun), gas, dust and dark matter, tied together by gravity. For example, we live in a galaxy that we like to call the Milky Way. They can have different shapes (or morphology). Astronomers like to classify them by shape. Two of such examples are shown in Figures 3 and 4.

Figure 4. The Large Magellanic Cloud : classified as an irregular galaxy (source: fromquarkstoquasars.com).

Organising your data

So, to organise your data, you could create a table like in Figure 5, listing all your galaxies and their respective morphology. You would then have a straight forward way to represent the information.

The problem with this is that if you delete all entries (rows), you would not only delete the objects, but the morphologies as well. You would therefore have to start from scratch.

Instead, we want to keep both of those entity type separate and create relations between them.

Using the principles of relational databases, the Galaxy and Morphology data can be restructured into two separate tables (say MORPH and GAL), as shown in Figures 6 and 7.

We now have created the relation between the tables, having the morphology data independent to the galaxy data, and associating each galaxy to a specific morphology.

Relation

From Figures 6 and 7, we can represent the relationship between MORPH and GAL such as depicted in Figure 8.
Here, each table is shown in a structural form as it would be represented in a digital database. The first row is the entity (the table name), the first column comprises the different field names, and the second column the relative field types shown in italic.
When the tables are implemented in the database, the information in the two tables is linked by using special columns called foreign keys. In our exemple, morphNo is our foreign key, and we represent the link with the straight line between the two tables and the key name.

Normalising the database

To avoid having inconsistencies in the database, where no undesired anomalies would occur during a modification, relational databases are normalised. As a rule of thumb, you can keep the following in mind when designing your first database (taken from here):

Table must be two-dimensional, with rows and columns.

Each row contains data related to one thing or one portion of a thing.

Each column contains data for a single attribute of the thing being described.

Each cell (intersection of row and column) of the table must be single-valued.

All entries in a column must be of the same kind.

Each column must have a unique name.

No two rows may be identical.

The order of the columns and of the rows does not matter.

Querying the database with SQL

Now that we have a structure for our database, it would be nice to be able to interact with it. For instance, it would be nice to add new instances of galaxies or morphologies. SQL is the language of (most) databases.
The SQL language consists of a only few key words. In this tutorial, we will only look at Query keywords shown on the right.

Querying the database with SQL

Here is a possible SQL query example applied to our previous example:
SELECT MORPH.morphName AS Name,
COUNT(*) AS Galaxies
FROM MORPH
JOIN GAL
ON MORPH.morphNo = GAL.morphNo
GROUP BY MORPH.morphName;

This query will return all morphological names from the table MORPH, and cound the number of individual galaxies that have this particular morphology as recorded in table GAL (Figure 9).

The beauty of SQL (I think!), is that it is very close to natural language. For instance, our query states almost literally: "please database, show me (or select) the records in column morphName from table MORPH and count how ofter they occur in table GAL. The two tables are joint through the columns morphNo of each table. Please group the results by morphName, so that I don't have to read the same value twice." Since there was only two records in Figure 7, the count of each morphology is 1.

Get crackin'

There are many digital database software out there. Some are easy to use, some are slightly more complicated. Some are free, some are proprietary. But they all have more or less the same basis, which you now know!

You can have a look at SQLite (which you may already have on your machine) and MySQL, which are both free and popular examples.

Most of them can be run from the terminal, but some also provide a Graphical User Interface (GUI). Have a look at the links from Figures 10 and 11. A GUI can be extremely useful, especially when you start dealing with relational databases. To start, I think SQLite with DB Browser (GUI) is a good place to start.

Here are some tutorials on MySQL and SQLite. And if you want to know more about SQL itself, I encourage you to look at w3school SQL's tutorial or at this quick reference page.

At this point in the workshop, I encourage you to give it a go for yourself. Try to start a database related to your own data. For instance, if you work in management, you could start a simple DB to list people by department (similarly to our galaxy by morphology).

Last but not least

Other topics you may want to read about (but not restricted to) are the different kind of keys (e.g. primary key), the different kind of joins (e.g. left join, right join, ...), how to add new records to one or multiple tables (INSERT INTO), etc.

Have a look at the left side bar of w3school It's a good source of information about SQL (and for all kind of web development).

You can then either use your database by itself to organise and mine your information, or plug it with a software or a website (e.g. Ruby on Rails, some Python web server of your choice or PHP).
